# CivicConnect AI - Firestore Database Design

This document outlines the Firestore NoSQL database schema for CivicConnect AI, designed for scalability, fast read operations, and efficient querying.

## 1. Users Collection
**Path:** `users/{userId}`
Stores citizen, admin, and official profiles.

**Fields:**
- `uid` (String, Document ID) - Firebase Auth UID
- `displayName` (String) - User's full name
- `email` (String) - Contact email
- `photoURL` (String) - Avatar image URL
- `role` (String) - `"citizen" | "official" | "admin"`
- `reputationScore` (Number) - Points earned for valid reports
- `createdAt` (Timestamp)
- `updatedAt` (Timestamp)

---

## 2. Reports Collection
**Path:** `reports/{reportId}`
The core collection storing civic issues reported by users. 

**Fields:**
- `reportId` (String, Document ID)
- `authorId` (String) - Reference to `users/{userId}`
- `title` (String) - AI-generated short title
- `description` (Map) - Multilingual descriptions generated by Gemini
  - `en` (String)
  - `es` (String)
  - `hi` (String)
- `category` (String) - e.g., `"Pothole", "Garbage", "Water Leakage"`
- `severity` (String) - `"Low" | "Moderate" | "Critical"`
- `status` (String) - `"Open" | "In Progress" | "Resolved" | "Duplicate"`
- `department` (String) - AI-recommended assigned department
- `imageUrl` (String) - Firebase Storage URL of the uploaded image
- `location` (GeoPoint) - Exact coordinates for Map rendering
- `address` (String) - Reverse-geocoded human-readable address
- `upvoteCount` (Number) - Denormalized counter for sorting/trending
- `commentCount` (Number) - Denormalized counter for UI display
- `isDuplicate` (Boolean) - Flag indicating if AI marked it as duplicate
- `createdAt` (Timestamp)
- `updatedAt` (Timestamp)
- `resolvedAt` (Timestamp, Optional)

---

## 3. Votes (Subcollection)
**Path:** `reports/{reportId}/upvotes/{userId}`
Tracks which users have upvoted a specific report. 

**Fields:**
- `userId` (String, Document ID) - Forces uniqueness (1 vote per user)
- `createdAt` (Timestamp)

---

## 4. Comments Collection
**Path:** `comments/{commentId}`
A root collection (rather than a subcollection) is preferred here so we can easily query "All comments by User X" across different reports, while still querying "All comments for Report Y".

**Fields:**
- `commentId` (String, Document ID)
- `reportId` (String) - Reference to `reports/{reportId}`
- `authorId` (String) - Reference to `users/{userId}`
- `authorName` (String) - *Denormalized* for faster UI rendering
- `authorPhoto` (String) - *Denormalized* for faster UI rendering
- `text` (String) - Comment content
- `createdAt` (Timestamp)

---

## 5. Notifications (Subcollection)
**Path:** `users/{userId}/notifications/{notificationId}`
Stores alerts for users (e.g., when their report is resolved). Stored as a subcollection because notifications are strictly private to the user.

**Fields:**
- `notificationId` (String, Document ID)
- `type` (String) - `"REPORT_STATUS_CHANGE" | "NEW_BADGE" | "MILESTONE"`
- `title` (String) - Notification headline
- `body` (String) - Details
- `linkTo` (String) - URL path to navigate to on click (e.g., `/reports/123`)
- `isRead` (Boolean) - `false` initially
- `createdAt` (Timestamp)

---

## 6. Badges Collection
**Path:** `badges/{badgeId}`
Global dictionary of available badges.

**Fields:**
- `badgeId` (String, Document ID) - e.g., `"first_report", "civic_hero"`
- `name` (String) - e.g., `"Civic Hero"`
- `description` (String) - e.g., `"Reported 50 verified issues."`
- `iconUrl` (String) - Image/SVG path

**User Earned Badges (Subcollection)**
**Path:** `users/{userId}/earnedBadges/{badgeId}`
- `badgeId` (String, Document ID) - Matches global badge ID
- `earnedAt` (Timestamp)

---

## 7. Leaderboard Collection
**Path:** `leaderboard/{userId}`
Instead of aggregating the `users` collection in real-time (which is expensive), we maintain a dedicated leaderboard collection updated via Cloud Functions.

**Fields:**
- `userId` (String, Document ID)
- `displayName` (String) - *Denormalized*
- `photoURL` (String) - *Denormalized*
- `totalPoints` (Number) - E.g., 10 pts per report, 5 pts per resolution
- `reportsSubmitted` (Number)
- `reportsResolved` (Number)
- `lastUpdated` (Timestamp)


## 🧠 NoSQL Best Practices Applied

1. **Denormalization for Fast Reads:**
   Notice that `authorName` and `authorPhoto` are stored directly on the `Comments` collection, and `Leaderboard` documents. In NoSQL, storage is cheap but queries are expensive. Denormalizing prevents the client from having to fetch the `users` document for every single comment rendered on the screen.

2. **Atomic Counters for Aggregations:**
   `upvoteCount` and `commentCount` are stored directly on the `Report` document. Instead of the client querying "Count all documents in the Votes subcollection", a Firebase Cloud Function (or the client itself) uses `FieldValue.increment(1)` to update the counter atomically whenever a new vote/comment is added.

3. **Strategic Subcollections vs Root Collections:**
   - **Upvotes:** Stored as a subcollection `reports/{id}/upvotes/{userId}`. By using the `userId` as the document ID, we inherently prevent duplicate voting (a user can't have two documents with the same ID in the same subcollection).
   - **Notifications:** Stored as `users/{userId}/notifications` because this data is deeply hierarchical and private to a specific user.
   - **Comments:** Stored as a root collection `comments`. While it *could* be a subcollection, making it a root collection makes it much easier to implement features like a "User Profile Activity Feed" (querying all comments made by `authorId == XYZ`, regardless of the report).

4. **Map Data Types for Localization:**
   The `description` field in `Reports` is a Map containing `en`, `es`, `hi`. This allows the UI to instantly pull the user's preferred language without extra queries, leveraging Gemini's translation output natively within a single document read.
